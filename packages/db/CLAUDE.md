# @strus/db — Agent Context

Drizzle ORM schema, migrations, and database client using the `bun:sqlite` adapter.

## Critical: Use `bun:sqlite`, NOT `better-sqlite3`

```ts
// ✓ correct
import Database from "bun:sqlite";
import { drizzle } from "drizzle-orm/bun-sqlite";

// ✗ wrong — better-sqlite3 is a Node.js native addon; Bun does not support it at runtime
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";
```

`better-sqlite3` is kept as a **devDependency** only because `drizzle-kit` (the schema diffing
CLI) requires it internally. It is never imported in any source file that runs under Bun.

## Schema File

`src/schema.ts` — single source of truth for all tables. Edit this to change the DB structure.

Current tables:
- `vocab_lists` — named collections of lemmas
- `lemmas` — dictionary citation forms; has `source` enum column (`"morfeusz"` | `"manual"`)
- `vocab_list_lemmas` — join table (listId, lemmaId)
- `morph_forms` — individual inflected forms generated by Morfeusz2
- `learning_targets` — FSRS card state per morphological tag
- `reviews` — immutable review history

## After Changing the Schema

```bash
# From the db package directory (or via repo root filter):
pnpm run generate
# or:
pnpm --filter @strus/db run generate
```

This runs `drizzle-kit generate` interactively. **It will ask about renames** — if you renamed
a table or column, select "rename" not "create+drop" to preserve existing data.

The generated SQL file lands in `migrations/`. **Commit it** — migrations are applied by the
API server at startup.

## Applying Migrations

Migrations are NOT applied by drizzle-kit at runtime. The API server applies them:

```ts
import { migrate } from "drizzle-orm/bun-sqlite/migrator";
migrate(db, { migrationsFolder: "/absolute/path/to/db/migrations" });
```

The migrations folder path must be **absolute** (use `resolve(import.meta.dir, "…")`).

## Query Style

`bun:sqlite` is **synchronous**. Drizzle queries return results directly:

```ts
// ✓ returning all rows
const rows = db.select().from(lemmas).all();

// ✓ returning first row (destructure)
const [row] = db.select().from(lemmas).where(eq(lemmas.id, id)).limit(1);

// ✓ insert
await db.insert(lemmas).values({ id, lemma, … });  // ok to await, resolves immediately

// ✓ aggregate
const [result] = await db.select({ value: count() }).from(lemmas);
const total = result?.value ?? 0;
```

## Column Type Reference

| Drizzle definition                        | TypeScript type          | SQLite type  |
|-------------------------------------------|--------------------------|--------------|
| `text("col").notNull()`                   | `string`                 | TEXT         |
| `text("col")`                             | `string \| null`         | TEXT         |
| `text("col", { enum: ["a","b"] })`        | `"a" \| "b" \| null`    | TEXT         |
| `integer("col", { mode: "timestamp" })`   | `Date`                   | INTEGER      |
| `integer("col").notNull().default(0)`     | `number`                 | INTEGER      |
| `real("col").notNull().default(0)`        | `number`                 | REAL         |
| `integer("col")` (nullable)               | `number \| null`         | INTEGER      |

Timestamp columns (`mode: "timestamp"`) are stored as Unix seconds and returned as `Date`
objects by Drizzle. To write: `new Date()`. To read: `.toISOString()` or `.getTime()`.

## drizzle-kit Commands

```bash
pnpm run generate   # generate migration from schema diff (interactive)
pnpm run migrate    # apply migrations via drizzle-kit (dev only; server does this at runtime)
```

The `migrate` script is for local dev convenience only. In production, the API server applies
migrations at startup via `drizzle-orm/bun-sqlite/migrator`.
